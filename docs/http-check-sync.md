# Синхронная HTTP-проверка

Синхронная проверка вызывает ваш сервис по URL из поля **`method`** тем же **POST**, что и асинхронная, но **результат ожидается в теле HTTP-ответа** этого вызова. Запись в **`product_ff`** выполняется сразу (логика совпадает с **`POST /api/v1/ff/webhook`**). В ответе **`POST /api/v1/run/{code}`** поле **`check_result`** содержит **`is_check`** из ответа сервиса.

## Когда выбрать этот режим

- Результат известен до завершения обработки запроса.
- Не нужен отдельный колбэк на webhook.

## Настройка в БД (признак режима)

- В таблице **`ff.fitness_function`**: непустой **`method`**, **`method_synchronous = true`**.

## Создание проверки через API

**`PUT /api/v1/fitness-function/{code}`** (`multipart/form-data`):

| Поле | Значение |
|------|----------|
| `description` | Описание |
| `method` | Полный URL (**POST**), куда FF Manager отправляет JSON |
| `method_synchronous` | **`true`** (или `1`, `yes`, `on` — см. обработку формы в сервисе) |

## Что отправляет FF Manager на ваш URL

Поведение запуска такое же, как у асинхронной проверки:

- **HTTP:** `POST`, `Content-Type: application/json; charset=utf-8`
- **Тело:**

```json
{
  "callId": "<uuid>",
  "productCode": "<мнемоника из поля app в run>"
}
```

### Параметр `docId`

Если при **`POST /api/v1/run/{code}`** или **`POST /api/v1/run-all`** в query указан **`docId`**, к URL из **`method`** добавляется **`docId=...`** (без изменения JSON-тела). Это работает и для синхронного, и для асинхронного режима.

## Ожидаемый ответ вашего сервиса (обязательно)

- Статус **2xx**.
- **Тело ответа — JSON-объект**, по смыслу как тело **`POST /api/v1/ff/webhook`**:

| Поле | Обязательность | Описание |
|------|----------------|----------|
| `callId` или `call_id` | Да | Должен **совпадать** с **`callId`**, который FF Manager отправил в запросе |
| `isCheck` или `is_check` | Да | Boolean — прошла ли проверка |
| `details` или `json_details` | Нет | Детали (строка, объект или массив) |
| `countDetail` / `count_detail`, `successDetail` / `success_detail` | Нет | Целые числа, если нужны |

Если **`callId`** в ответе не совпадает с исходящим, JSON невалиден, нет **`isCheck`**, или **`isCheck`** не boolean — вызов считается ошибкой, ожидающая запись вызова удаляется, **`run`** вернёт ошибку.

Таймаут запроса к вашему URL: **`FF_EXTERNAL_FF_TIMEOUT_SECONDS`**.

## Что происходит после успешного ответа

1. Результат записывается в **`product_ff`** (как при webhook).
2. Контекст вызова в **`outside_ff`** помечается завершённым.
3. Клиент **`run`** получает **`check_result`**: значение **`is_check`** из JSON-ответа.

## Запуск проверки

- **`POST /api/v1/run/{code}`** — `{"app": "<мнемоника>"}`; опционально **`?docId=...`**
- **`POST /api/v1/run-all`** — аналогично; проверки со статусом **TEST** в run-all не участвуют

Учитываются правила **`applicability`** (не для **TEST**; **TRIAL**/**ADOPT** — как боевая проверка).

## Сравнение с асинхронной HTTP-проверкой

| | Синхронная | Асинхронная |
|---|------------|-------------|
| **`method_synchronous`** | `true` | `false` |
| Результат | В **теле ответа** на первый POST | Через отдельный **`POST /api/v1/ff/webhook`** |
| **`check_result` в ответе `run`** | Обычно задан (`is_check`) | Обычно **`null`** до webhook |

Подробности исходящего запроса и **`docId`**: см. также **`docs/http-check-async.md`**.
