# Асинхронная HTTP-проверка

Асинхронная проверка вызывает ваш сервис по URL из поля **`method`**, затем результат записывается отдельным запросом **`POST /api/v1/ff/webhook`** (колбэк). До прихода webhook в **`product_ff`** нового результата нет; в ответе **`POST /api/v1/run/...`** поле **`check_result`** обычно **`null`**.

## Когда выбрать этот режим

- Долгая или фоновая логика на стороне внешнего сервиса.
- Результат готов не в момент HTTP-ответа, а позже.

## Настройка в БД (признак режима)

- В таблице **`ff.fitness_function`**: непустой **`method`**, **`method_synchronous = false`** (значение по умолчанию для HTTP-проверок).

## Создание проверки через API

**`POST /api/v1/fitness-function`** (`multipart/form-data`):

| Поле | Значение |
|------|----------|
| `code` | Уникальный код проверки |
| `description` | Описание |
| `method` | Полный URL, на который FF Manager шлёт **POST** (JSON) |
| `method_synchronous` | Не передавать, или **`false`** / не отмечать (как принято в вашем клиенте для «ложного» флага) |

Поле `method_synchronous` в форме распознаётся как логическое: истинными считаются, например, `true`, `1`, `yes`, `on` (см. логику в коде). Для асинхронного режима флаг должен быть **ложным**.

Список проверок и поля можно посмотреть: **`GET /api/v1/fitness-functions`**.

## Что отправляет FF Manager на ваш URL

После **`POST /api/v1/run/{code}`** или попадания проверки в **`POST /api/v1/run-all`**:

- **HTTP:** `POST` на URL из **`method`**
- **Заголовок:** `Content-Type: application/json; charset=utf-8`
- **Тело JSON:**

```json
{
  "callId": "<uuid — идентификатор вызова>",
  "productCode": "<мнемоника продукта из тела run: поле app>"
}
```

Имена полей в исходящем теле: **`callId`**, **`productCode`** (camelCase).

### Параметр `docId`

Если при запуске в query передан **`docId`** (к **`/api/v1/run/{code}`** или **`/api/v1/run-all`**), к URL из **`method`** добавляется query-параметр **`docId=...`** (существующие параметры в URL сохраняются). Тело JSON при этом **не меняется**.

## Ожидаемый ответ вашего сервиса на первый POST

- Код **2xx** — вызов принят; контекст вызова сохраняется, ожидается webhook.
- Любой **не-2xx** или сетевой сбой — вызов считается неуспешным, запись ожидающего вызова удаляется; **`run`** вернёт ошибку.

Таймаут исходящего запроса: **`FF_EXTERNAL_FF_TIMEOUT_SECONDS`** (по умолчанию 30 с), см. `config.py` / переменные окружения.

## Колбэк с результатом: `POST /api/v1/ff/webhook`

Ваш сервис должен вызвать FF Manager (базовый URL должен быть доступен из вашей сети):

**`POST {FF_API_BASE_URL}/api/v1/ff/webhook`**

Тело JSON (допустимы синонимы полей, как в OpenAPI):

| Смысл | Варианты имён |
|--------|----------------|
| Идентификатор вызова | `callId` или `call_id` — **тот же**, что пришёл в исходящем POST |
| Успех проверки | `isCheck` или `is_check` (boolean) |
| Детали | `details` или `json_details` (строка, объект или массив) |
| Счётчики | `countDetail` / `count_detail`, `successDetail` / `success_detail` |

Повторный webhook с тем же `callId`, если статус вызова уже **готов**, обрабатывается как «уже обработано» (идемпотентность на стороне API).

## Запуск проверки

- **`POST /api/v1/run/{code}`** — тело: `{"app": "<мнемоника>"}`; опционально query **`docId`**
- **`POST /api/v1/run-all`** — то же; проверки со статусом **TEST** в run-all не входят

Перед запуском учитывается **`applicability`**, кроме статуса **TEST** (результат в **`product_ff`** не сохраняется). **TRIAL** и **ADOPT** ведут себя одинаково.

### Тестовая асинхронная проверка (`status = TEST`, `method_synchronous = false`)

1. **`POST /api/v1/run/{code}`** — в **`check_result`**: `{ "pending": true, "callId": "<uuid>" }`.
2. Ваш сервис отвечает **2xx**, затем вызывает **`POST /api/v1/ff/webhook`** с тем же **`callId`**.
3. **`GET /api/v1/ff/call/{callId}`** — пока webhook не пришёл: **`status": "pending"`**; после: **`status": "done"`** и **`check_result`** с **`is_check`**, **`details`**, счётчиками (без записи в **`product_ff`**).

Для отладки без ожидания webhook можно временно включить **`method_synchronous = true`** — тогда детали сразу в ответе **`run`**.

## Краткий сценарий

1. Создать проверку с **`method`** и **`method_synchronous` = false**.
2. Вызвать **`run`** — FF Manager POST’ит на **`method`** с **`callId`** и **`productCode`**.
3. Ваш сервис отвечает **2xx**, асинхронно считает результат.
4. Ваш сервис вызывает **`/api/v1/ff/webhook`** с тем **`callId`** и **`isCheck`** / деталями.
5. Результат появляется в **`product_ff`**; **`GET /api/v1/product/{code}/actual-results`** начнёт его отдавать (если проверка не вспомогательная и не тестовая).
